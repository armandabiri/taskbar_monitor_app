# Plan

## 1. SYSTEM OVERVIEW

This repository is a Windows-only PyQt6 desktop application that combines a taskbar resource monitor with process cleanup utilities, snapshot capture, clipboard history, and native toast notifications.

Scope and boundaries:

- The UI shell lives in `src/main.py` and the dialogs under `src/ui/`.
- Cleanup behavior is implemented in `src/services/resource_control/`.
- Snapshot capture, storage, and diff logic live in `src/services/process_snapshot.py` and the snapshot dialogs.
- Persistence uses `QSettings` for preferences and CSV files under `QStandardPaths.AppDataLocation` for snapshots.
- Notifications are delivered through `src/services/notification_service.py`.
- The app does not run a server, expose a network API, or rely on any remote service.

Key components:

- `src/main.py`: main window, button wiring, release triggers, notification entrypoints, logging setup.
- `src/services/resource_control/service.py`: orchestrates scan, trim, throttle, kill, and system-wide reclaim steps.
- `src/services/resource_control/candidate_scorer.py`: filters and scores live processes for trim, throttle, and kill eligibility.
- `src/services/resource_control/profiles.py`: built-in presets and persisted custom profile state.
- `src/services/resource_control/windows_ops.py`: Win32 and psutil process operations, visible-window detection, tray enumeration, termination, working-set trimming.
- `src/services/process_snapshot.py`: snapshot CSV capture, load, list, rename, delete, and snapshot-to-snapshot diff logic.
- `src/ui/snapshot_manager_dialog.py`, `src/ui/snapshot_diff_dialog.py`, `src/ui/kill_confirm_dialog.py`: snapshot workflow and destructive-action confirmation.
- `src/services/notification_service.py`: thin wrapper over `win11toast`.

## 2. CURRENT STATE ANALYSIS

What exists:

- The app already supports profile-driven cleanup with trim, throttle, kill, standby flush, working-set empty, and modified-page flush.
- The app already supports taking process snapshots, storing them as CSV, comparing two snapshots, and invoking cleanup from a selected snapshot.
- The kill path already has a confirmation dialog and basic protection rules for visible windows, tray icons, protected names, protected users, and Windows system binaries.

What is missing:

- There is no dedicated "snapshot extra processes" execution path. Snapshot cleanup is routed through the generic resource reclaim engine instead of an explicit process-diff workflow.
- There is no persistent or user-visible reason breakdown when a cleanup run does nothing.
- There is no single authoritative "processes cleaned" metric in the result model or the notifications.
- There is no live snapshot-to-current-system preview that shows which processes were added since the snapshot and which of them are actually actionable.
- There is no automated test suite for candidate filtering, snapshot matching, zero-action summaries, or snapshot cleanup behavior.

What is weak or problematic:

- R1 root cause: `src/main.py` routes snapshot cleanup through `release_resources(profile=profile, snapshot_spare_keys=...)`. That design does not mean "kill extra processes." It means "run the normal cleanup algorithm, and spare anything whose `(name, exe)` appears in the snapshot." The cleanup result stays empty whenever all new processes are protected by the normal safety rules or filtered out before kill selection.
- R1 root cause: kill is disabled in the default aggressive preset. `DEFAULT_AGGRESSIVE_NAME` points to `Aggressive`, and `AGGRESSIVE.enable_kill` is `False`. Users can press the aggressive button and never enter the kill phase.
- R1 root cause: even in `Nuclear`, `spare_visible_windows=True` and `spare_tray_icons=True`. Any newly opened app with a visible window or tray icon is silently excluded from kill, which makes snapshot-driven "remove extras" behavior ineffective for common desktop apps.
- R1 root cause: kill eligibility is constrained by same-user ownership, process age, protected names, protected users, Windows directory exclusion, foreground exclusion, visible-window sparing, and tray sparing. These are correct safety checks, but the app does not show which rule rejected each process.
- R2 root cause: `ReleaseResult.summary` only reports successful trim, kill, and throttle counts. It does not report skipped candidates, blocked candidates, snapshot extras found, snapshot extras spared, or why zero actions happened.
- R2 root cause: logging writes only aggregate text such as `1 errors` without the full reason summary. The current log cannot answer why a cleanup run produced zero actions.
- R4 root cause: `diff_against_live()` already exists in `src/services/process_snapshot.py`, but nothing calls it. The repository contains the start of a snapshot-to-live feature, but the actual UI and execution path were never finished.
- R4 root cause: `ProcessSnapshot.spare_keys()` collapses the snapshot into a set of `(name, exe)` pairs. That destroys instance counts and command-line identity. If one `chrome.exe` exists in the snapshot, every later `chrome.exe` with the same executable path is spared. This makes "kill extra processes" impossible for duplicate instances.
- The snapshot manager text promises behavior that is broader than what the engine actually does. The UI presents "Clean now" as a direct action, but the underlying implementation remains profile-dependent and safety-filter-dependent.
- `src/taskbar_monitor.log` is written under `src/`, which is a poor runtime location for packaged builds and for end-user diagnostics.
- There is no cleanup history, no dry-run preview, and no explicit post-run explanation path for partial failures such as `AccessDenied`.

Helpful functionality missing around the same use case:

- A live preview of added processes since a snapshot, with per-row memory size, executable path, command line, visibility state, tray state, and default selection state.
- A "why nothing happened" result view that lists the top rejection reasons instead of only showing zero successful actions.
- A cleanup history view that lets the user inspect the last runs, their counts, the dominant skip reasons, and the first execution errors.
- A user-managed keep list for apps that must never be killed even in snapshot-extra mode.

## 3. PROPOSED ARCHITECTURE

Target system design:

- Keep the existing profile-based resource reclaim flow for normal smart and aggressive cleanup.
- Add a second cleanup mode named `snapshot_extras` that is independent from memory-pressure planning and operates on exact live PIDs that were added after a chosen snapshot.
- Unify both modes behind a richer cleanup result model so the UI, notifications, and logs always explain what was found, what was selected, what ran, what was blocked, and what failed.

Module and component breakdown:

- `src/services/resource_control/models.py`
  - Add `CleanupScope`, `SkipReason`, `CandidateDecision`, `CleanupHistoryEntry`, and expanded `ReleaseResult` fields.
  - Track `processes_cleaned_total`, `kill_candidates_found`, `snapshot_extras_found`, `snapshot_extras_selected`, `blocked_reason_counts`, `executed_reason_counts`, `memory_before_gb`, and `memory_after_gb`.
- `src/services/resource_control/candidate_scorer.py`
  - Return a structured decision object instead of only `ProcessCandidate | None`.
  - Preserve the exact rejection reason for every filtered process.
- `src/services/resource_control/service.py`
  - Split the work into `scan`, `plan`, `execute`, and `summarize`.
  - Support two scopes: `system_reclaim` and `snapshot_extras`.
  - In `snapshot_extras`, bypass the memory-pressure short-circuit and skip trim/throttle selection entirely unless the user explicitly invoked the normal reclaim flow.
- `src/services/resource_control/snapshot_scope.py` (new)
  - Convert a stored snapshot plus current live processes into an exact set of extra live PIDs.
  - Use multiset matching so only the excess instances are treated as extras.
- `src/services/process_snapshot.py`
  - Add normalized process identity helpers that use executable path, process name, username, and command line.
  - Keep the existing CSV schema and derive identities from the fields already stored.
- `src/ui/snapshot_live_cleanup_dialog.py` (new)
  - Show the live diff against a chosen snapshot and let the user approve exact extra processes before execution.
- `src/ui/cleanup_result_dialog.py` (new)
  - Provide a compact explanation dialog for zero-action and partial-failure runs.
- `src/services/resource_control/history.py` (new)
  - Persist a bounded JSONL history of cleanup runs for diagnostics and future UI use.

Data flow:

1. Normal cleanup:
   - `src/main.py` builds `CleanupScope(mode="system_reclaim")`.
   - `service.py` samples memory before the run, scans processes, records every accept/reject decision, executes trim/throttle/kill, samples memory after the run, and returns a fully populated `ReleaseResult`.
2. Snapshot-extra cleanup:
   - `src/ui/snapshot_manager_dialog.py` loads the chosen snapshot and opens `snapshot_live_cleanup_dialog.py`.
   - `snapshot_scope.py` computes exact extra live PIDs by comparing a multiset of normalized identities from the snapshot to a multiset of normalized identities from current live processes.
   - The preview dialog marks hidden/background extras as selected by default and marks visible-window or tray-resident extras as unselected by default, with explicit badges explaining why.
   - `service.py` executes kill only for the PIDs that the user approved.
3. Reporting:
   - `ReleaseResult` drives toast text, button tooltip text, the optional result dialog, the persistent history file, and the log output.

Technology choices:

- Continue using Python, psutil, PyQt6, and `win11toast`.
- Add `pytest` and `pytest-qt` to the dev dependencies for automated coverage of service and dialog behavior.
- Keep snapshot storage as CSV because the current fields already contain enough identity material; add derived matching logic instead of introducing a migration.

## 4. INTERFACES & CONTRACTS

Core service contracts:

- `build_process_identity(name: str, exe: str, username: str, cmdline: str) -> tuple[str, str, str, str]`
  - Returns a normalized identity tuple used for snapshot/live matching.
  - Normalization rules:
    - lowercase all string parts
    - trim whitespace
    - normalize empty values to `""`
    - collapse repeated whitespace inside command lines

- `diff_snapshot_to_live(snapshot: ProcessSnapshot, live_processes: Iterable[psutil.Process] | None = None) -> SnapshotLiveDiff`
  - Input: a stored snapshot and optionally a provided live-process iterator for tests.
  - Output:
    - `extra_processes: list[LiveSnapshotExtra]`
    - `matched_count: int`
    - `missing_snapshot_count: int`
    - `identity_collisions: int`
  - Matching algorithm:
    - build a `Counter` of normalized snapshot identities
    - build a live list grouped by the same identity
    - mark only the surplus live instances as extras
    - order surplus instances by newest `create_time` first, then highest RSS

- `plan_cleanup(profile: ResourceProfile, scope: CleanupScope, confirm_kill: ConfirmKillCallback | None = None) -> ReleaseResult`
  - `scope.mode="system_reclaim"`: current trim/throttle/kill behavior plus reason tracking.
  - `scope.mode="snapshot_extras"`: kill only the PIDs in `scope.target_pids`; do not use reclaim thresholds to suppress the run.

- `CandidateDecision`
  - Required fields:
    - `pid`
    - `name`
    - `candidate: ProcessCandidate | None`
    - `eligible_for_trim: bool`
    - `eligible_for_throttle: bool`
    - `eligible_for_kill: bool`
    - `skip_reason: SkipReason | None`
    - `snapshot_status: str`

- `SkipReason`
  - Required enum values:
    - `OWN_PROCESS`
    - `PROTECTED_NAME`
    - `PROTECTED_USER`
    - `WINDOWS_BINARY`
    - `FOREGROUND_PROCESS`
    - `VISIBLE_WINDOW`
    - `TRAY_ICON`
    - `NEW_PROCESS_GRACE`
    - `DIFFERENT_USER`
    - `RECENTLY_TRIMMED`
    - `RECENTLY_THROTTLED`
    - `BELOW_TRIM_THRESHOLD`
    - `NO_RECLAIM_VALUE`
    - `SNAPSHOT_BASELINE_MATCH`
    - `ACCESS_DENIED`
    - `EXECUTION_FAILED`

Release result contract:

- `ReleaseResult.summary`
  - First line format:
    - normal cleanup: `Cleaned X process(es): Y trimmed, Z killed, W throttled`
    - snapshot cleanup: `Snapshot extras: F found, S selected, K killed`
  - Zero-action runs must include the dominant block reason:
    - `No processes cleaned. Blocked mostly by visible-window protection (5).`

- `ReleaseResult.details`
  - Must include:
    - memory before and after
    - total candidates scanned
    - kill candidates found
    - snapshot extras found and selected when applicable
    - top five skip reasons with counts
    - first five execution errors

Notification contract:

- `NotificationService.notify_cleanup(mode_name: str, result: ReleaseResult) -> bool`
  - Returns `True` on success and `False` on toast failure.
  - Uses the new `summary` as the toast title/body pair instead of raw free-form strings from `main.py`.

## 5. DATA & STATE MANAGEMENT

Data models and schemas:

- Keep the existing snapshot CSV schema in `src/services/process_snapshot.py`.
- Add derived runtime models:
  - `LiveSnapshotExtra`
    - `pid`
    - `name`
    - `exe`
    - `username`
    - `cmdline`
    - `rss_gb`
    - `create_time`
    - `has_visible_window`
    - `has_tray_icon`
    - `default_selected`
    - `default_block_reason`
  - `CleanupHistoryEntry`
    - `run_id`
    - `timestamp`
    - `mode`
    - `profile_name`
    - `snapshot_name`
    - `processes_cleaned_total`
    - `processes_trimmed`
    - `processes_killed`
    - `processes_throttled`
    - `kill_candidates_found`
    - `snapshot_extras_found`
    - `blocked_reason_counts`
    - `errors`

Storage strategy:

- Continue storing snapshots under `QStandardPaths.AppDataLocation/snapshots`.
- Add `cleanup_history.jsonl` under the same writable app-data root.
- Move `taskbar_monitor.log` to the writable app-data root and stop writing runtime logs into `src/`.
- Continue using `QSettings` for profile bindings and add explicit keys for snapshot-preview defaults.

State handling:

- `ActivityTracker` remains in-memory for normal reclaim runs.
- Snapshot-extra mode does not depend on `ActivityTracker` cooldown state to decide whether a selected extra process should be killable.
- After a successful kill, remove the PID from in-memory trim/throttle caches so stale state does not affect later runs.
- Persist only user preferences and cleanup history. Do not persist live process decisions between runs.

## 6. EXECUTION FLOW

Normal cleanup flow:

1. User clicks the smart or aggressive button in `src/main.py`.
2. `main.py` resolves the active profile and builds `CleanupScope(mode="system_reclaim")`.
3. `service.py` samples memory before execution and scans live processes.
4. Every scanned process receives a `CandidateDecision`, including an explicit skip reason when rejected.
5. The planner selects trim and throttle targets from accepted candidates.
6. If kill is enabled, the service produces a kill candidate list and passes it to `confirm_kill`.
7. The executor runs the approved actions, records per-action success or failure, and samples memory after execution.
8. `main.py` shows a toast using `NotificationService.notify_cleanup`, updates the button tooltip with observed freed memory, and opens `cleanup_result_dialog.py` when the run was zero-action or partially failed.

Snapshot-extra cleanup flow:

1. User opens the snapshot manager and clicks `Kill extras...` for a saved snapshot.
2. `snapshot_manager_dialog.py` loads the snapshot and opens `snapshot_live_cleanup_dialog.py`.
3. `snapshot_scope.py` compares the stored snapshot identities to current live identities and returns only the extra live PIDs.
4. The preview dialog renders every extra process with RSS, executable path, command line, visibility state, tray state, and default selection.
5. Hidden/background extras are checked by default.
6. Visible-window and tray-resident extras are shown but unchecked by default, with explicit warning labels.
7. If the user confirms, `main.py` calls `plan_cleanup(..., scope=CleanupScope(mode="snapshot_extras", target_pids=...))`.
8. `service.py` kills only the selected PIDs, records successes, `AccessDenied` failures, and "already exited" outcomes, then returns a snapshot-specific `ReleaseResult`.
9. The app shows a snapshot-specific toast and records the run to history.

Zero-action flow:

1. The service returns `processes_cleaned_total == 0`.
2. `ReleaseResult.summary` states `No processes cleaned`.
3. `ReleaseResult.details` lists the top rejection reasons and the number of processes in each category.
4. `cleanup_result_dialog.py` opens automatically when the run was user-triggered from the UI.

## 8. TESTING & VALIDATION

Unit testing strategy:

- Test every `SkipReason` path with synthetic `psutil.Process` doubles so a zero-action run always produces a stable and explainable reason summary.
- Test `ReleaseResult.summary` and `ReleaseResult.details` string rendering for:
  - successful reclaim runs
  - zero-action runs
  - partial failures
  - snapshot-extra runs
- Test `diff_snapshot_to_live()` with:
  - one-to-one matches
  - duplicate instances
  - missing executable paths
  - empty command lines
  - mixed usernames

Integration testing:

- Run the app locally on Windows and capture a baseline snapshot.
- Start multiple disposable user-owned processes such as `notepad.exe` and a sleeping Python script, then verify that the snapshot preview reports only the processes that were started after the baseline.
- Verify that if the snapshot contains one instance of an app and the live system contains three identical instances, the preview shows exactly two extras.
- Verify that visible-window extras appear unchecked by default, while hidden/background extras appear checked by default.
- Verify that a normal cleanup run below the pressure threshold still reports `No processes cleaned` with the correct dominant block or suppression reason instead of only showing zeros.
- Verify that an `AccessDenied` target increments the failure count, appears in the result dialog, and is written to the log and history.

Validation criteria:

- Snapshot-extra mode kills only the exact approved extra PIDs.
- Zero-action notifications always include a reason summary.
- The cleanup log always records the top skip reasons for each run.
- The history file contains one record per run and honors retention limits.
- Existing smart and aggressive cleanup behavior remains intact for non-snapshot flows.

## 10. RISKS & LIMITATIONS

- Process-identity matching is heuristic when `exe` or `cmdline` is empty. The multiset matcher reduces false positives, but identical processes with missing metadata can still collapse into the same identity. The preview dialog must show those rows explicitly before kill.
- Visible-window and tray-icon detection on Windows 11 is not perfect. The new design must treat those signals as user-facing safety hints, not as silent hard filters in snapshot-extra mode.
- Killing user-owned applications can destroy unsaved work. The new snapshot-extra mode must always require an explicit preview and confirmation before execution.
- Memory reclaimed after trim or kill is not deterministic because Windows may defer reclamation. The result model must report observed before/after memory, not promise a fixed reclaim amount.
- `AccessDenied` and transient process exits are unavoidable on Windows. The plan addresses this by reporting them precisely and keeping the run partially successful instead of hiding them behind aggregate zero counts.
